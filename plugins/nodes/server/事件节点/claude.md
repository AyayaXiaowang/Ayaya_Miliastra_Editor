# Server 事件节点目录

## 目录用途
- 服务器侧事件节点定义，一节点一文件，对应运行期触发点（定时器、背包、实体、巡逻等）。
- 事件函数只负责调度同一事件流中的后继节点，不直接执行复杂业务。

## 当前状态
- 由静态注册器纳入 `plugins.nodes.server`，供 `from plugins.nodes.server import *` 统一导入。
- 节点函数均使用 `@node_spec` 描述端口，依赖共用 helper 位于 `plugins/nodes/shared`.
- 部分关键语义事件节点会在 `@node_spec` 中声明 `semantic_id`（稳定语义标识符），供校验/工具链定位语义节点，避免依赖显示名字符串（例如信号监听、自定义变量变化事件等）。
- 信号相关事件节点 `监听信号` 通过一个字符串输入端口“信号名”选择要监听的信号，并在引擎信号系统提供的参数定义基础上，在 UI 中为每个信号参数自动追加对应的数据输出端口。

## 注意事项
- 保持文件名与事件名称一致，便于查找与生成文档。
- 不要在事件节点里导入 UI 层或自动化模块，仅依赖 `engine` 与 `plugins.nodes.shared`.
- 输出日志统一通过 `engine.utils.logging.logger`.
- 若事件节点的输出端口类型为“枚举”，应在 `@node_spec` 中补齐 `output_enum_options`，以支持编辑器下拉选择与校验层对枚举字面量做严格约束。
- 常见需要显式候选项的事件输出包括：倒下原因、元素类型、实体类型、比较运算类型等；候选项以业务约定为准，禁止用模糊描述或占位字符串。
- 事件节点的输出端口名会直接作为 Graph Code 的 Python 形参名：必须是合法标识符；若历史端口名包含 `：`/`/` 等符号，应改为下划线命名，并用 `output_port_aliases` 兼容旧图（必要时同步保留旧端口名的枚举候选集合，避免候选集合推断退化）。
- 业务语义明确且键/值类型固定的字典输出端口（如：商店收购事件的“收购物品字典”）统一使用 `配置ID-整数字典`，避免用裸 `字典` 放大类型歧义。
