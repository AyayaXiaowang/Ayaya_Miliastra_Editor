# Server 查询节点目录

## 目录用途
- 集中声明服务器侧“查询类”节点：读取仇恨值、背包、装备、环境配置、路径点等。
- 每个节点函数返回计算或查询结果供后续节点使用。

## 当前状态
- 所有节点以 `@node_spec` 定义端口，必要的常用逻辑放入 `plugins.nodes.shared.server_查询节点_impl_helpers`。
- 部分关键语义查询节点会在 `@node_spec` 中声明 `semantic_id`（稳定语义标识符），供校验/工具链定位语义节点，避免依赖显示名字符串（例如节点图变量读取、自定义变量读取等）。
- 由静态注册表暴露给 `plugins.nodes.server`.
 - 支持少量“工程化查询”入口（例如 UI 交互：事件源GUID → action_key/args），用于将外部资源映射注入到节点图的可读分发逻辑中。
- 查询节点实现函数签名统一包含 `game` 作为第一个参数（即使节点本身不使用运行态），以匹配 Graph Code/导出代码的统一调用形态：`节点函数(self.game, ...)`。
 - 本地测试（MockRuntime）下，部分查询节点提供“最小可用语义”用于跑通图逻辑：例如 `获取在场玩家实体列表` 读取 `GameRuntime` 的在场玩家配置并返回稳定实体列表（用于多人等待/投票门槛模拟）。
 - 本地测试下，`获取玩家当前界面布局` 会读取运行态的 `ui_current_layout_by_player`，用于支持“仅在某布局内处理点击”的门控逻辑（例如第七关结算页按钮处理）。
 - 本地测试下，时间/计时器相关查询（如 `查询游戏已进行时间/查询当前环境时间/获取全局计时器当前时间`）基于 `time.monotonic()` 与 `game.timers` 提供可执行语义，便于倒计时/昼夜门控等逻辑在离线环境跑通。
 - 本地测试下，职业/成就等状态类查询优先从自定义变量读取（如 `当前职业配置ID/职业等级_<id>/成就进度计数_<idx>`），作为业务系统未接入时的最小承载。
 - 本地测试下，纯常量/工具类查询节点（如 `圆周率/三维向量常量/查询时间戳/查询服务器时区/列表工具`）同样保持 `game` 作为第一个参数，避免源码/导出代码的调用形态分叉。
 - 本地测试下，实体类型相关查询支持多来源解析：实体属性 `entity_type` / 自定义变量 `实体类型` / 基于实体名称的启发式推断（玩家/角色/造物/关卡/物件）。
 - 本地测试下，范围筛选基于实体 `position` 计算；阵营筛选优先读取实体属性 `camp` 或自定义变量 `阵营`。
 - 本地测试下，向前/向右/向上向量基于实体 `rotation`（欧拉角 `[pitch, yaw, roll]`）计算（最小语义忽略 roll）。
 - 本地测试下，预设状态与部分属性类查询允许通过自定义变量注入（变量名与端口显示名一致；未设置时回退为 0/0.0）。
 - 本地测试下，拥有者关系查询优先读取实体 `owner` / `owner_entity_id`，并允许通过自定义变量 `owner_entity_id` 注入。
 - 本地测试下，`查询自定义变量快照` 以 dict 作为快照承载结构进行查询。

## 注意事项
- 查询节点只读状态，不在内部修改 GraphModel 或运行态数据。
- 输出日志统一走 `engine.utils.logging.logger`，避免私有 print。
- 共通格式化/校验逻辑尽量抽到 shared helper，保持节点文件简洁。
- 若输入/输出端口类型声明为“枚举”，应在 `@node_spec` 中补齐 `input_enum_options`/`output_enum_options` 候选项集合，避免编辑器侧下拉缺项。
- 常见需要显式候选项的查询输出包括：结算状态、输入设备类型、游玩方式、受打断状态等；候选项以业务约定为准，禁止用无意义占位字符串绕过校验。
- 常见需要显式候选项的查询输入包括：实体类型筛选、结算状态筛选等；输入端若声明了候选项，Graph Code 传入的字符串字面量必须落在候选集合内。
- 端口方向必须与语义一致：输入端口用于提供查询条件，输出端口用于返回查询结果；函数返回值的结构与顺序应与 `@node_spec(outputs=...)` 保持一致，避免 Graph Code/校验层出现“端口方向反了”的隐性错误。
- 字典查询节点的字典端口统一使用“泛型字典”端口类型；键端口通过 `input_generic_constraints` 声明仅接受实体、GUID、整数、字符串、阵营、配置ID、元件ID，值相关输出端口不允许字典类型，约束列表中也不要放入泛型或泛型列表。
- 业务语义明确且键/值类型固定的字典输出端口（如：货币/道具数量字典、出售货币字典、收购货币字典）统一使用 `配置ID-整数字典`，避免用裸 `字典` 放大类型歧义。
- 【获取局部变量】的“初始值/值”端口禁止字典类型（含别名字典与泛型字典）；仅允许基础类型/列表类型/枚举/结构体等，避免把字典当作可持久化局部状态使用。
