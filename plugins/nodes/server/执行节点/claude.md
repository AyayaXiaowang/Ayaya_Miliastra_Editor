# Server 执行节点目录

## 目录用途
- 服务器执行节点的集中地，涵盖战斗、背包、计时器、巡逻等“动作”指令。
- 每个 `.py` 文件实现一个节点函数，通过流程图调度触发。

## 当前状态
- 所有函数使用 `@node_spec` 声明端口类型与相关元数据（包含泛型约束与枚举候选项等），与 `plugins.nodes.shared.server_执行节点_impl_helpers` 等辅助库配合。
- 部分节点会在 `@node_spec` 中声明 `input_defaults`（输入端口默认值），用于让特定数据输入端口在 Graph Code/图模型中允许省略；对应实现函数需提供匹配的 Python 默认参数值，确保运行期调用不因缺参报错。
- `创建元件` 的 `旋转` 端口允许省略：通过 `input_defaults` 提供缺省值，Graph Code 不需要显式传入旋转。
- 部分关键语义节点会在 `@node_spec` 中声明 `semantic_id`（稳定语义标识符），供校验/工具链定位语义节点，避免依赖显示名字符串（例如信号发送、自定义变量写入、结构体修改等）。
- `发送信号`（`semantic_id="signal.send"`）在本地测试（MockRuntime）中提供可执行语义：以 `信号名` 作为事件名触发 `game.trigger_event(...)`，并自动补齐 `事件源实体/事件源GUID/信号来源实体`；动态端口参数以关键字参数形式原样并入事件上下文。
- 节点库构建由 V2 AST 管线扫描 `@node_spec`（只解析不导入）；运行期需要执行节点函数时，由 `app.runtime.engine.node_impl_loader` 按需加载并注入。
- `创建实体`/`创建元件` 等“生成类”执行节点会提供“创建后实体”数据输出端口，便于后续节点直接以该实体作为数据输入连线。
- 部分与界面布局、角色技能相关的执行节点（例如切换当前界面布局、设置角色技能冷却等）依赖精确定义的枚举输入，便于在编辑器中通过下拉选项选择布局索引或技能槽位，而无需手填魔法数字或字符串。
- 字典工具类执行节点（清空/设置/移除/排序）提供可执行语义：对传入 `dict` 原地修改；排序节点返回 `键列表/值列表` 并支持 `排序规则_顺序/排序规则_逆序`，便于本地测试闭环（MockRuntime）复用。
- “角色技能槽位”等枚举端口的候选项应完整声明（普通攻击/技能1-4 + 自定义技能槽位1-15），避免不同节点的下拉选项不一致。
- 卡牌选择器、掉落与结算相关执行节点也依赖稳定的枚举输入（如刷新方式、掉落类型、结算状态等），必须在 `@node_spec` 中补齐 `input_enum_options`，禁止使用无意义占位字符串绕过校验。
- 跟随运动器/单位状态/排序等执行节点同样依赖明确的枚举输入（坐标系类型、跟随位置类型、单位状态移除策略、排序规则等），必须声明候选项以启用严格校验与 UI 下拉。
- 战斗表现相关执行节点（如损失生命、扫描标签规则等）也需要声明枚举候选项（伤害跳字类型、规则类型等），避免节点图写入自由字符串导致不可控的运行期分支。
- 包含用于排查逻辑的轻量调试类执行节点（如打印字符串），用于在运行日志中输出自定义文本。
- 运行时事件触发口径：本目录内执行节点在需要广播事件时，统一使用 `game.trigger_event(event_name, **kwargs)`，并使用对应事件节点的端口名作为参数键，避免位置参数导致协议漂移。
- 本地测试（MockRuntime）支持：
  - 界面相关执行节点（`切换当前界面布局/修改界面布局内界面控件状态/激活控件组库内界面控件组/移除控件组库内界面控件组`）会在运行时调用 `GameRuntime` 的 UI 模拟接口，记录 UI patch 供浏览器预览回显显隐/状态切换（不代表真实游戏 UI）。
  - 定时器相关执行节点（`启动定时器/终止定时器`）会在运行时调用 `GameRuntime.start_timer_sequence/stop_timer`，并由本地测试轮询驱动 `tick()` 触发事件 `定时器触发时`，用于倒计时/状态机推进。
  - 基础运动器相关执行节点（`开启定点运动器/停止并删除基础运动器`）在本地测试下提供可执行语义：通过 `GameRuntime.start_motor/stop_motor` 启动/停止离线运动器模拟，并由 `tick()` 到期触发事件 `基础运动器停止时`，用于验证“运动器结束→信号/状态机推进”的主链路（节点图无需用定时器兜底迎合测试）。
  - 定时器暂停/恢复执行节点（`暂停定时器/恢复定时器`）在本地测试下通过标记 `game.timers[timer_key]["paused"]` 并平移 `start_time` 实现冻结/恢复计时（`next_fire_time` 置空以让 `tick()` 跳过）。
  - `设置节点图变量` 统一委托 `GameRuntime.set_graph_variable(...)`，与 `获取节点图变量` 的读路径对齐，确保离线运行时变量读写一致。
  - 结构体修改（`修改结构体`）在本地测试下对 `dict` 结构体就地更新字段，并维护 `__field_order` 元数据，保证后续拆分/再次修改的字段顺序稳定。
  - 排行榜/成就/职业等状态类执行节点在本地测试下提供“最小可用语义”：以自定义变量与 `game.leaderboards` 作为存储承载，便于跑通图逻辑与事件流联动。

## 注意事项
- 仅依赖 `engine` 与 `plugins.nodes.shared`，避免引用 UI/自动化模块。
- 输出日志、调试信息统一走 `engine.utils.logging.logger`。
- 触发运行时事件统一使用 `game.trigger_event(event_name, **kwargs)` 的关键字参数形式，避免位置参数导致协议不匹配（离线 MockRuntime 同样遵循该约定）。
- 调整端口（inputs/outputs）或函数 `return` 行为后，建议跑一次节点图校验与 pytest 回归，并在 UI 中打开相关节点图确认端口与连线恢复正常。
- 复杂的复用逻辑抽到 shared helper，保持节点文件聚焦于参数校验与调用顺序。
- 执行节点的流程口（如“流程入/跳出循环”）用于图结构连线，不要求出现在函数签名中；函数签名中的 `*_callback/*_callbacks` 参数用于表达流程输出分支的可执行回调（同样不视为输入端口）。
- 端口名是图模型与 Graph Code 的**权威标识**：若端口名包含括号/斜杠等无法作为关键字参数名的字符，则在 Graph Code/导出可执行代码时应使用**位置参数**传值（避免生成语法错误）。
- 节点的端口命名需与底层图模型/代码生成管线约定保持一致，避免出现端口存在但类型信息无法匹配的情况；尤其是信号与结构体相关执行节点，需要包含用于选择资源的名称端口，其名称与 `engine.graph.common` 中的静态端口常量保持一致。
  - 结构体相关执行节点【修改结构体】不再声明独立的“结构体名”端口：仅保留一个结构体端口（端口名统一为`结构体`，仅允许连线），结构体名仅作为 `node.input_constants["结构体名"]` 展示/兼容常量存在；旧端口名 `结构体实例` 通过端口别名兼容并在 UI 预处理阶段自动迁移为 `结构体`。
- 字典端口类型约定：
  - 通用字典工具节点（清空/查询/排序/拼装/建立等）使用“泛型字典”端口类型；若存在独立的键/值端口，则在 `@node_spec` 中通过 `input_generic_constraints` 声明具体允许类型：键仅支持实体、GUID、整数、字符串、阵营、配置ID、元件ID；值不允许字典类型；约束列表使用具体数据类型，不要放入泛型或泛型列表。若节点仅暴露单一“字典”端口（无独立键/值端口），可直接对该端口用 `input_generic_constraints` 枚举允许的“别名字典类型”（`键类型-值类型字典`），用于限制键/值范围。
  - 业务语义明确且键/值类型固定的字典输入端口，优先使用“别名字典”类型名（`键类型-值类型字典` / `键类型_值类型字典`，例如：`配置ID-整数字典`），避免用裸 `字典/泛型字典` 放大类型歧义。
  - 商店（出售/收购）等“货币字典”输入端口统一使用 `配置ID-整数字典`：键为货币配置ID，值为整数数量/价格，避免编辑器端口类型不明确。
