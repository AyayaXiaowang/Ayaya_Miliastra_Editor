# Client 查询节点目录

## 目录用途
- 客户端查询节点：读取本地状态、局部变量、扫描结果、实体属性等数据。

## 当前状态
- 节点函数均使用 `@node_spec`，共享逻辑由 `plugins.nodes.shared.client_查询节点_impl_helpers` 管理。
- 已提供常用查询节点（如获取当前角色、获取在场玩家实体列表、获取自定义变量等），供技能节点图与本地过滤器节点图读取所需上下文数据。
- 已提供字典查询节点【以键查询字典值】（用于语法糖 `值 = 字典.get(键)`）。
- 部分关键语义查询节点会在 `@node_spec` 中声明 `semantic_id`（稳定语义标识符），供校验/工具链定位语义节点，避免依赖显示名字符串。
- 通过静态注册器向 `plugins.nodes.client` 暴露。
 - 本地测试（MockRuntime）下，`获取在场玩家实体列表` 会读取 `GameRuntime` 的在场玩家配置并返回稳定实体列表，便于复用同一套多人逻辑在本地跑通。

## 注意事项
- 查询节点只读取客户端可见数据，避免直接访问服务器全局状态。
- 输出调试信息统一使用 `engine.utils.logging.logger`。
- 如需复杂数据处理，优先放在 shared helper，再在节点中引用。
- 列表相关查询节点统一使用“泛型列表”端口类型表达输入容器类型，元素端口通过泛型约束声明具体允许的数据类型，避免在 Graph Code 中出现结构体或其他非容器类型误接到列表端口。
- 若输出端口类型声明为“枚举”，应在 `@node_spec` 中补齐 `output_enum_options`，避免编辑器侧下拉缺项，并支持校验层对枚举字面量做严格约束。
- 若输入端口类型声明为“枚举”，应在 `@node_spec` 中补齐 `input_enum_options`，以支持射线/筛选等查询节点的阵营筛选、实体类型筛选等下拉选项与严格校验。
- 多选筛选项应使用 `枚举列表`（而非单个 `枚举`），并为列表元素提供候选项（例如射线检测的命中层筛选），避免“单选枚举”误建模为多选列表或反之。
