## 目录用途
`engine/graph/ir/` 负责把类结构 Graph Code 的 AST 转换为 `GraphModel`：控制流建模、节点/端口构造、参数归一化、变量环境与流程/数据连线路由。

## 当前状态
- **入口**：`flow_builder.parse_method_body` + `flow_utils.materialize_call_node` 负责“语句 → 节点/边”的主流程；节点创建在 `node_factory.create_node_from_call`。
- **稳定 NodeDef 引用**：IR 产出的所有 `NodeModel` 都必须填充 `node_def_ref`（`NodeDefRef(kind+key)`）作为 NodeDef 定位的唯一真源；后续校验/UI/自动化/导出不得再基于 `title` 做 NodeDef 查找。
- `node_factory.create_node_from_call` 支持在“嵌套调用”场景识别复合节点实例方法调用（`self.<复合实例>.<入口>(...)`），使其可作为其它节点的入参数据来源，避免父节点出现“输入端缺少数据来源”。
- `return` 语句在 IR 层默认被视为“终止当前语句块解析”：遇到 `return` 会停止继续解析后续语句，避免把不可达语句错误连到图中；分支出口是否可接续由 `branch_builder.block_has_return` 判定，**仅将分支体顶层 `return` 视为“该分支整体终止”**（嵌套在 match/if 内的 return 不会让外层分支体整体失去接续能力）。对 **client 过滤器节点图**（布尔/整数）则会把 `return <值>` 物化为对应的【节点图结束（布尔型/整数）】纯数据节点，并将返回值通过连线/常量回填到其输入端 `结果`。
- **常量体系（已下沉）**：
  - 模块级命名常量通过 `ast_utils.collect_module_constants` 设置为上下文，供 `extract_constant_value` 解析 `ast.Name`。
  - 方法体内命名常量预扫描写入 `VarEnv.local_const_values`，节点创建时可将 `变量名` 实参回填为 `node.input_constants`。
  - IR 在写入 `node.input_constants` 时保留原始 Python 值类型（bool/int/float/str/None/容器），不在构建阶段统一 `str()`；避免数字常量“字符串化”导致 GraphModel 有效类型推断漂移（例如 `拼装列表 → 浮点数列表`）。
- **可选输入端口默认值**：当节点定义（`NodeDef`）声明了 `input_defaults` 时，IR 在创建 NodeModel 时会为未显式传入的对应端口写入默认常量（`input_constants`），从而在结构校验与 UI 中表现为“可不提供来源但仍有缺省值”。
- **类型追踪（轻量）**：
  - `VarEnv.var_types` 记录变量中文类型注解与少量容器节点（如【拼装列表/拼装字典/建立字典】）的类型占位推断，用于局部变量建模策略与跨块解析一致性。
  - 局部变量建模（`local_var_builder`）在生成【获取局部变量/设置局部变量】节点时，会把已知的变量类型写入 `GraphModel.metadata["port_type_overrides"]`（覆盖 `初始值/值` 等端口），确保画布端口类型展示与有效类型推断不被“常量字符串化”误导。
- **作用域上下文**：`FactoryContext.graph_scope` 由解析入口写入（server/client），供 IR 子模块在同名节点存在 `#client/#server` 变体且端口不兼容时做语义分支（例如：局部变量建模在 client 下使用【获取局部变量(变量名→变量值)】+【设置局部变量(变量名/变量值)】，server 下使用【获取局部变量(初始值→局部变量/值)】+【设置局部变量(局部变量/值)】）。
- **语义元数据（单点写入）**：
  - IR 层不再写入 `GraphModel.metadata["signal_bindings"/"struct_bindings"]`；这些字段统一由 `engine.graph.semantic.GraphSemanticPass` 覆盖式生成（幂等、可复现）。
  - IR 层只负责把“选择端口常量/动态字段端口”建模到 NodeModel（`input_constants` + `ports`），作为 Pass 的推导输入。
  - 结构体语义节点兼容：Graph Code 仍可传入 `结构体名="xxx"` 作为展示/绑定线索，但该字段不再建模为真实输入端口（结构体节点仅保留一个结构体端口），仅写入 `input_constants["结构体名"]`。
- 复合节点虚拟引脚构建以**类格式**为前提（pin_marker + 方法签名推断），不再支持旧函数式复合节点“按顶层函数签名提取引脚”的路径，避免解析口径分裂。
- 复合节点数据入引脚类型合并规则：方法签名的类型标注作为默认类型；只有当方法体内 `数据入(...)` 显式提供了**非“泛型”**的 `pin_type` 时，才会覆盖签名类型，避免“写了数据入但未写类型”把签名标注错误覆盖为“泛型”而导致 UI 显示异常。
- 事件处理方法参数名推导会对事件节点输出端口名做 `make_valid_identifier` 规范化并保证唯一，避免端口名包含 `：`/括号等导致生成代码出现非法标识符。
- 循环节点（`有限循环` / `列表迭代循环`）由 IR 建模为 **执行节点**（对齐节点库分类），端口名与节点定义保持一致（例如 `列表迭代循环` 的列表入参为 `迭代列表`），以保证端口类型推导与缓存一致性。

## 注意事项
- 端口/参数归一化必须通过 `arg_normalizer.normalize_call_arguments`，避免端口名分叉。
- 输出变量绑定（`register_output_variables`）必须以“上下文感知的流程口判定”为准：优先通过 node_library 的端口类型字典识别流程口，避免把“完成/成功/失败”等命名流程口误当成数据输出端口，导致严格结构校验出现“流程口→数据口”的错误连线。
- 对结构体语义节点【拆分结构体】的动态输出端口名：若赋值变量名以“字段”或“字段_后缀”结尾（如 `进度_当前关卡ID字段`、`武器条目ID字段_控件`），端口名会自动去掉该尾缀以对齐真实结构体字段名与 `struct_bindings`。
- 本目录保持纯逻辑（不做 UI）；`FactoryContext` 仅承载节点库与名称索引（`node_library/node_name_index`），不再承担语义元数据推导所需的索引注入。
- “纯别名赋值”（`目标 = 某变量`）默认可通过 `handle_alias_assignment` 走快速路径，但当目标变量处于“多分支合流/已有局部变量句柄”模式时必须禁用该优化，避免出现“生成获取局部变量但缺少设置局部变量”的异常图。
  - 带类型注解的“纯别名赋值”（`别名: "类型" = 原变量`）会写入 `GraphModel.metadata["port_type_overrides"]`：用于将事件/信号等声明为“泛型”的输出端口在具体图中实例化为明确类型，避免画布与结构校验出现“泛型未实例化”。
  - 带类型注解的“常量占位初始化”（例如 `x: "整数" = 0`）若会在被读取前被后续赋值覆盖（常见于“为解包赋值预声明类型/默认值”的写法），解析会将其视为**声明/类型提示**并跳过建模，避免在画布上生成大量零散的【获取局部变量】节点。
- IR 解析期会区分“提示/警告”与“无法可靠建模的错误”：`Validators` 同时收集 `warnings` 与 `errors`，上层可在严格模式下将 `errors` 视为 fail-closed 的拒绝条件（拒绝产图，避免静默产错图）。
- 循环（for）体解析会对 `VarEnv` 做 snapshot/restore：循环体内对同名变量的重复赋值（典型：累计/计数）会自动建模为【获取局部变量】初始化 + 循环内【设置局部变量】写回；一旦变量已进入【获取/设置局部变量】建模（句柄存在），后续赋值必须继续生成【设置局部变量】写回，避免跨迭代/跨块状态更新丢失。
- 分支/循环解析的 `VarEnv.snapshot/restore` 必须同时恢复变量映射与类型映射，避免块内推断污染外层判断。
- 局部变量建模策略：
  - 字典类型（含别名字典/泛型字典）禁止转换为【获取/设置局部变量】。
  - 列表类型在“由【拼装列表】构造且后续存在原地修改节点”时，会自动插入一次【获取局部变量】固定列表引用，避免拉取式重复求值导致每次修改落在新列表实例上。
  - 对“先初始化、后在 if/match 分支体内再次赋值、且分支后继续使用”的变量，解析会将初始化提前建模为【获取局部变量】以提供稳定句柄，避免出现“句柄只在某个分支内首次创建、另一分支只能引用该句柄”的错误图结构。
  - 多分支（match-case）分支体在解析时使用“分支端口作为前驱”（如 `(多分支节点, "0")`），确保 break 可在无前置流程节点时正确连到循环的【跳出循环】，并允许“分支体仅含纯数据节点”时从分支端口继续接续后续流程。
  - 复合节点 match（`match self.<复合实例>.<入口>(...)`）同样以“复合节点的流程出口端口作为前驱”（如 `(复合节点, "出口名")`）解析 case 分支体：保证循环内 break 的连线不丢失，并支持纯数据/空分支继续接续；同时对所有有效 case 的赋值交集推导 multi-assign 提示以支持跨分支合流。
- 复合节点实例方法调用（`self.<复合实例>.<入口>(...)`）在 IR 物化时同样支持“参数内嵌套节点调用”（如 `右值=加法运算(...)`）：需要先提取嵌套节点并建立其数据边，再将嵌套节点输出连接到复合节点输入端口，避免出现“输入端缺少数据来源”。
- 本文件不记录修改历史，只描述“目录用途、当前状态、注意事项”的实时概况。
