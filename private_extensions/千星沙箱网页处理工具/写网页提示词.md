一、 核心技术限制
技术栈：仅限 HTML + CSS。
你要写的是一个纯静态展示用的网页，没有任何交互。所有的按钮都只能看，不管用户如何要求，你都是写纯静态的。

❌ 禁止 JS。
文件结构：单文件 HTML。
分辨率适配：必须支持以下 4 种尺寸，且无滚动条：

1920 × 1080

1600 × 900

1560 × 720

1280 × 720

补充：文件落点与动态文本占位符（必须遵守）
- HTML 源码保存位置：`assets/资源库/项目存档/<项目>/管理配置/UI源码/`
  - 约定：一个 HTML 文件对应一个界面布局（一个网页 = 一个 UILayout）
- 动态文本必须用占位符声明（否则一律视为静态文本）：
  - `{{lv.<变量名>}}`：关卡实体自定义变量（来自当前项目引用的关卡变量文件或共享变量文件）
  - `{{ps.<变量名>}}`：玩家模板自定义变量（**只允许**玩家模板 `metadata.custom_variable_file`；**禁止** `metadata.ingame_save_variable_file`）
  - `{{<变量名>}}`：兼容旧写法；若变量在 lv/ps 同时存在会要求显式写前缀
- 占位符里的 `<变量名>` 必须与变量定义中的 `variable_name` 完全一致；导入到项目存档/生成 GIL 时会做强校验，不存在就会报错中止。
- 颜色也按“文本”处理：你可以把 `<color=#RRGGBB>...</color>` 或 `#RRGGBB` 当作变量值写进自定义变量，然后在 HTML 文本里直接插入对应占位符。

二、 视觉与图层构成规范（关键修正）

1. 阴影（两种用途，必须分清）

1.1 投影阴影 (Drop Shadow)
- **CSS 实现**：使用 `box-shadow`。
- **强制参数**：
  - 颜色 + 透明度：**必须写成带 alpha 的颜色**（45% 透明度），禁止“纯黑/纯不透明阴影”
    - ✅ 推荐：`rgba(14, 14, 14, 0.45)`
    - ✅ 允许等价：`#0E0E0E73`
    - ❌ 禁止：`#000` / `#000000` / `#0E0E0E` / `rgb(...)`（不带 alpha）
  - 模糊 (Blur)：`0px`
  - 扩散 (Spread)：`0px`
- **示例**：`box-shadow: 5px 5px 0px rgba(14, 14, 14, 0.45);`

1.2 盖色阴影（Shade Overlay，用于“压暗/按下态/调亮度”）
- **目的**：在固定色板的基础色上“盖一层 45% 阴影色”，得到更暗的按下态颜色。
- **CSS 实现**：用**真实 DOM 子元素**做覆盖层（不要用 `::before/::after`），放在**底色之上、文字之下**。
- **覆盖层颜色**：`background-color: rgba(14, 14, 14, 0.45);`
- **推荐写法**（示意，按你的布局体系自行套类名）：
  - 容器（通常是 `<button>`）：`position: relative;`
  - 盖色层：`position: absolute; inset: 0; background: rgba(14,14,14,0.45);`
  - 文本：放在盖色层后面（DOM 顺序在后），或显式 `z-index` 更高

2. 边框层 (Border Layer)
CSS 实现：使用 border 属性。

规范：

必须是实线 (solid)。

颜色必须取自指定色板。

3. 主体层 (Body Layer)
CSS 实现：使用 background-color 属性。

转化逻辑：去除边框后的内部背景区域。

规范：

必须设置非透明背景色（尤其是有边框的文本标签）。

4. 文本层 (Text Layer)
CSS 实现：元素的 innerText 以及 padding。

转化逻辑：覆盖在最上方，保留原始 padding、对齐方式和文本样式。

规范：

Padding：必须显式管理 padding，因为它定义了文本层的点击/交互区域大小。

溢出处理：默认单行 (white-space: nowrap)，特定区域白名单允许对应多行。

三、 布局与层级堆叠 (Z-Index)
为了配合工具的扁平化覆盖逻辑，CSS 的层级管理必须清晰：

自然堆叠顺序：

遵循 DOM 树顺序：子元素 > 父元素。

后一个兄弟元素 > 前一个兄弟元素（及其子树）。

内部层级逻辑：

对于同一个 HTML 元素，视觉上的覆盖关系永远是：文本 > 主体 > 边框 > 阴影。

编写 CSS 时注意：不要使用伪元素 (::before/::after) 去模拟会导致层级混乱的背景或遮罩，直接使用标准的盒模型属性 (border, background, box-shadow)，以便工具按上述顺序提取。

绝对定位慎用：

尽量使用 Flexbox 布局。如果必须使用 absolute，必须确保其 z-index 不会破坏上述的自然堆叠逻辑，防止转化后图层错乱。

四、 颜色板 (严禁例外)
背景/边框/阴影颜色只能来自以下集合（严禁例外）；**文本颜色允许任意**（不做色板限制）：

4.1 基础色 (Base)
- 绿色：`#92CD21`
- 白色：`#E2DBCE`
- 黄色：`#F3C330`
- 蓝色：`#36F3F3`
- 红色：`#F47B7B`

4.2 阴影色
- 45% 阴影盖色：`rgba(14, 14, 14, 0.45)`（用于 `box-shadow` 或“盖色阴影”）
- 25% 阴影盖色：`rgba(14, 14, 14, 0.25)`（用于“更轻的盖色阴影”）
- ❌ 禁止在非文本颜色中使用不透明近黑（例如 `#0E0E0E` / `#000` / `#000000`）

4.3 压暗 1 级色（灰 1 级）
- 定义：在“基础色（Base）”上叠加 **一层** `rgba(14, 14, 14, 0.25)`。

  - 暖灰褐（白色压暗 1 级）：`#ADA89E`
  - 古铜/赭石（黄色压暗 1 级）：`#BA9628`
  - 深青（蓝色压暗 1 级）：`#2CBABA`
  - 铁锈红（红色压暗 1 级）：`#BA6060`

4.4 压暗 2 级色（原“压暗色”）
- 定义：在“基础色（Base）”上叠加 **一层** `rgba(14, 14, 14, 0.45)`。
- 颜色（允许在 background/border/shadow 的 hex 中直接使用）：
  - 深橄榄绿（绿色压暗 2 级）：`#577718`
  - 暖灰褐（白色压暗 2 级）：`#837F78`
  - 古铜/赭石（黄色压暗 2 级）：`#8C7221`
  - 深青/孔雀蓝（蓝色压暗 2 级）：`#248C8C`

4.5 压暗 3 级色（原“压暗二级色”）

补充：导出/写回规则
- 压暗 1 级（4.3）会被归一化为：Base + 1 层 25% 盖色阴影。
- 压暗 2 级（4.4）会被归一化为：Base + 1 层 45% 盖色阴影。

五、 游戏区域（挖空 / 漏出游戏内容）
- 用途：声明“这里是游戏画面露出来的区域”，背景层在扁平化/导出时会自动把该区域挖空（把一个大矩形切成多个小矩形围住它）。
- 写法（一个区域一个元素，同页允许多个）：
  - `class="game-cutout"`
  - 可选命名：`data-game-area="主视口"`（仅用于调试/检查器显示）
- 约束（强制）：
  - 必须是**方形**（宽高相等）
  - 必须是**透明**（background-color: transparent）
  - 不允许边框/阴影/文字（border/box-shadow/text）

六、 多状态控件（按钮/面板的 enabled/disabled/选中/未选中 等）
- 目标：在“纯静态 HTML+CSS”里表达“同一个控件有多种状态”，并让 Workbench 导出时保留两套状态内容，初始只显示默认态；运行时由节点图负责切换显隐。
- 写法（强约束）：
  - 用最近的容器声明状态组（同一组=同一个逻辑控件）：
    - `data-ui-state-group="<组名>"`：必填，同组的不同状态必须一致（建议用英文/拼音，如 `StartButton` / `Tab_1`）
    - `data-ui-state="<状态名>"`：必填，如 `enabled` / `disabled` / `selected` / `normal`
    - `data-ui-state-default="1"`：仅默认态填写（同组只允许一个默认态）
  - 每个状态的“完整内容”必须放在各自的状态容器里（不要试图只改颜色/只改一行文字）。
  - **禁止 `display:none` 隐藏非默认态**：display:none 会导致元素没有盒子，Workbench 无法提取坐标与导出。
  - 隐藏非默认态请用：`visibility:hidden`（推荐）或 `opacity:0` 且 `pointer-events:none`。

六点五、 选中高亮 / 按下态 / 列表互斥（统一用 data-ui-state-*）
- 目标：对“列表选中/Tab 选中/按下态”等互斥视觉，统一使用 `data-ui-state-group / data-ui-state / data-ui-state-default` 表达；运行时由节点图切换显隐。
- ❌ 禁止：`data-ui-selected-highlight` / `data-ui-selected-default`（已废弃，避免引入第二套状态体系）。
- 推荐写法（示意：同一个“选中状态组”，每个按钮内放一层高亮底板层；只切高亮层显隐，其余按钮结构保持一致）：

```html
<button type="button" data-ui-key="level_01" data-ui-role="button" data-ui-interact-key="1">
  <span class="level-highlight"
        data-ui-key="level_highlight"
        data-ui-state-group="level_selected"
        data-ui-state="level_01"
        data-ui-state-default="1"></span>
  <span class="label">第一关</span>
</button>

<button type="button" data-ui-key="level_02" data-ui-role="button" data-ui-interact-key="2">
  <span class="level-highlight"
        data-ui-key="level_highlight"
        data-ui-state-group="level_selected"
        data-ui-state="level_02"></span>
  <span class="label">第二关</span>
</button>
```

- CSS 要点：
  - 非默认态初始隐藏：`visibility:hidden`（推荐）或 `opacity:0 + pointer-events:none`
  - ❌ 禁止 `display:none`

```html
<!-- 逻辑控件：StartButton -->
<div class="ui-state" data-ui-state-group="StartButton" data-ui-state="enabled" data-ui-state-default="1">
  <!-- enabled 态：完整按钮（底色/边框/文字/装饰等） -->
</div>

<div class="ui-state" data-ui-state-group="StartButton" data-ui-state="disabled">
  <!-- disabled 态：完整按钮 -->
</div>
```

- 推荐 CSS（示意）：

```css
.ui-state { position: absolute; left: 0; top: 0; }
.ui-state:not([data-ui-state-default="1"]) { visibility: hidden; pointer-events: none; }
```

七、 可交互按钮（强制规范）
- 目标：让 Workbench **稳定**把“可点击控件”导出为 **道具展示（可交互锚点）**，避免因为 class 命名不同导致漏转。
- 强约束（优先级从高到低）：
  1. **优先使用 `<button type="button">`** 来写任何“看起来是按钮/需要点击”的控件（哪怕页面是静态稿）。
  2. 若因结构原因必须用 `div/span`，则必须显式标注其语义为按钮（二选一即可）：
     - `role="button"`（标准 ARIA）
     - `data-ui-role="button"`（工具私有约定，推荐）
  3. 兼容旧写法：`class` 中包含 **精确 token** `btn`（例如 `class="btn xxx"`）也会被当作按钮锚点；但不推荐依赖这一条（容易写成 `btn-text` 导致误判/漏判）。
- 注意：
  - 静态稿仍然 **不允许 JS**；这里的 `<button>`/role 仅用于“导出时识别交互语义”，不代表网页里真的要交互。
  - 不要给非按钮元素随便加 `role="button"` / `data-ui-role="button"`，否则会被导出为可交互锚点，影响后续打组与节点图逻辑。

八、 UI 交互动作标注（推荐，用于节点图逻辑解耦）
- 目标：让程序员在节点图里用“事件源GUID → 动作(action_key/args)”的方式写逻辑，而不是手工数索引或硬编码一堆 GUID 分支。
- 核心原则：
  - HTML 只标注“这次点击代表什么动作”，**不绑定实现方式**（不强制发送信号；也可以结束游戏/杀人/死亡等）。
  - 写回 `.gil` 时会自动生成一份映射文件：把“可点击道具展示”的 **GUID** 映射到 `action_key/action_args/ui_key`，供程序员查阅或自动生成节点逻辑。
- 写法（加在可点击按钮根元素上，推荐与 `data-ui-key` 一起使用）：
  - `data-ui-action="<动作ID>"`：例如 `ui.select_level` / `ui.start_challenge` / `ui.exit`
  - `data-ui-action-args="<参数文本>"`：参数原样透传（建议用 JSON 或 `k=v;k2=v2` 简写）
- 示例：

```html
<button type="button"
        data-ui-key="关卡按钮_第一关"
        data-ui-action="ui.select_level"
        data-ui-action-args='{"level": 1}'>第一关</button>
```

- 生成文件位置（项目存档内）：
  - `管理配置/UI交互映射/<ui_bundle_json文件名>.ui_actions.json`

九、 UI 交互按键码标注（可选；强约束：同页≤14且不重复）
- 背景：游戏内“可交互道具展示”需要绑定“奇匠按键码”。手柄侧只有 1..14 共 14 个按键槽位，因此为了保证“键鼠/手柄都能操作且一一对应”，同一个页面（一个 HTML / UILayout）内：
  - 可交互按钮数量 **不能超过 14**
  - 每个按钮必须占用 **不同的按键码（1..14）**
- 写法（加在按钮根元素上）：
  - `data-ui-interact-key="1"` ~ `"14"`
- 规则：
  - 若你显式写了 `data-ui-interact-key`，工具会按该值写入 `.gil`（道具展示的键鼠/手柄按键码会同时写成同一个数字）。
  - 若你没写，工具会按“页面顺序（从上到下、从左到右）”自动分配未占用的 1..14。
  - 若出现重复/超范围/同页超过 14 个，导出/写回会直接报错中止（避免生成游戏内不可用的存档）。
- 示例：

```html
<button type="button"
        data-ui-key="关卡按钮_第一关"
        data-ui-action="ui.select_level.1"
        data-ui-interact-key="1">第一关</button>
```

十、 按钮“装备配置ID变量”（自动填充；无需真实道具存在）
- 背景：可交互按钮锚点会被导出为“道具展示”，并且它需要绑定一个“装备配置ID变量”（挂在玩家自身自定义变量上）作为可写载体。
- 约定：所有按钮共用同一个变量名，不需要每个按钮单独一份，也不要求该配置ID真的对应一个存在的道具。
- 默认值：
  - 若你不写任何标注，工具会自动绑定到：`玩家自身.UI_交互按钮_装备配置ID`，并在写回 `.gil` 时自动创建该玩家自定义变量（类型=配置ID，默认值=0）。
  - 若你希望自定义变量名，可以在 UI JSON 或后续扩展里覆盖，但一般不推荐（增加维护成本）。
