## 目录用途
- UI 侧“执行子系统”：把任务清单（Todo）步骤序列转为对真实编辑器的自动化操作，并在右侧监控面板中展示截图/日志/调试叠加。
- 本目录负责执行编排与 UI 展示；底层截图/OCR/键鼠输入由 `app.automation` 提供并通过协议接口访问。

## 当前状态
- **入口**：
  - `runner.py`：执行入口与生命周期管理（执行整图/事件流/单步等）。
  - `planner.py`：将当前 Todo 上下文规划为可执行步骤序列。
  - `thread.py`：后台顺序执行步骤（缩放确认→映射→逐步执行→重试/跳过策略），并与监控面板联动输出日志与截图。
  - `guides.py`：对不支持自动化或只需提示的步骤输出“只读指引”。
  - `execution_session.py`：执行会话载体（workspace/model/view/executor 等上下文）。
  - `editor_executor_provider.py`：执行器获取/复用策略的统一入口，避免 UI 各处自行创建。
- **策略类**：`strategies/` 承载锚点选择、映射协调、跳过判定、重试与摘要构建等可组合策略（详见 `strategies/claude.md`）。
- **监控面板**：`monitor/` 承载监控面板本体与截图/日志/定位等辅助模块（详见 `monitor/claude.md` 与 `monitor/recognition_actions/claude.md`）。

## 注意事项
- **职责边界**：执行线程不负责节点图业务语义，只关心“如何按顺序调用执行器与策略对象”；资源/存档/图编辑语义应留在上层控制器与模型层。
- **异常可见**：不做静默兜底；错误直接抛出或通过监控面板日志明确暴露，便于定位。
- **协议访问**：与自动化执行器的交互必须通过协议/公开方法（缩放检查、识别映射、执行单步等）；不要在 UI 侧直接调用底层截图或输入实现。
- **可维护性**：新增执行阶段优先扩展/新增 `strategies/` 策略，再在 `thread.py` 中以清晰阶段顺序组合调用，避免单文件继续膨胀。
